4. Issues Tracker

  • A new Issues Tracker has been added to better keeps track of bugs and feature requests. ;The Issues Tracker home page can be accessed by clicking on the Bugs icon.

  • Terminologies used in the Issues Tracker:

    • Issue Type: Can be either a Defect, Enhancement or Maintenance.

      • Defect A bug in the source code.

      • Enhancement Feature request/improvement suggestion.

      • Maintenance System administration/maintenance request.

    • Severity: How severe the issue is and to what degree the system is affected by the issue.

      • For defects:

        • Fatal: The defect causes the system to crash and not operate at all. For example:

          • An infinite loop in the code causing the web page forever to load.

          Major: The defect impacts the operation/functionality of the system in major ways. For example:

          • An error in the SQL INSERT query causing Lab administrators cannot create new libraries.

          Minor: The defect impacts the operation/functionality of the system in minor ways. For example:

          • A bug in the JavaScript code causing the radio buttons in the No Grows/Slow Grows page not able to batch select/deselect checkboxes.

          • After scanning in a stock catalog number, the New Stock form does not have information pre-filled.

          Cosmetic: The defect does not impact the operation/functionality of the system in any ways. It is merely a cosmetic/user-interface problem. For example:

          • Spelling mistakes.

          • he layout on the webpage is skewed-up, even though all the functionalities/features can still be accessed.

      • For enhancements:

        • Fatal: The enhancement have a system-wide effect or requires a significant amount of code to be rewritten. In general the risk will be high; the enhancement will require a fair amount of work and extremely vigourous testing, and failure to do so will likely have a fatal effect on the system if the enhancement is made into the production release. An extensive design/code review with another developer is required. For example:

          • Implements MySQL transactions.

          Major: The enhancement affects a relatively large portion of the system. In general there will be a fair amount of risks involved; the enhancement will require a fair amount of work and very throughout testing. There should be at least brief design/code review with another developer. For example:

          • Implements the new generate_form() routine for generating HTML forms.

          Minor: The enhancement affects a small portion of the system. In general there will be little risks involved; the enhancement should be relatively easy to be implemented and relatively throughout testing will be required. For example:

          • Adding the ability in the Plates home page to find plates by rack.

          Cosmetic: The enhancement just affects the visual appearance of the system. ;In general there should be almost no risk involved; the enhancement should be extremely easy to be implemented and minimal testing will be required. For example:

          • Change the toggling colour in the HTML forms to improve the forms apperance.

      • For Maintenance

        • Fatal: The maintenance will cause the lab not to be able to operate for a period of time and there is no other way around this. These type of maintenance must be planned carefully ahead of time to minimize the systems down-time. For example:

          • Migrating the alDente system from web03/athena to seqdb01.

          Major: The maintenance will have major impacts on the operation of the lab, although the lab should still be fully functional. These type of maintenance is somewhat risky as well. For example:

          • paraAltering the indexing of the Clone_Sequence table.

          Minor: The maintenace will have minor impacts on the operation of the lab. For example:

          • Moving trace files from mirror folder into the archive folder.

          Cosmetic: The maintenance should have no effects on the operation of the lab.

          • Cleanup up the temp files in /home/sequence/ to save some space.

    • Status: The current status of the issue.

      • Reported: The issue has been submitted but the developers haven't looked into the issue yet.

      • Open: The developers are aware of the issue and is currently addressing it. Note that an issue will be in this state as long as the developers are still working on a resolution.

        • For defects, the issue is expected to be resolved in the current production release.

        • For enhancements, the issue is expected to be resolved in the next production release.

      • Deferred: The developers are aware of the issue. Note that issues that are deffered cannot have a priority of 'Critical' or 'High'. (see above)

        • For defects, the issue will be addressed in the next production release or any future production releases.

        • For enhancements, the issue will be addressed in any future production releases (i.e. not the next production release).

      • Deferred: The developers are aware of the issue. Note that issues that are deffered cannot have a priority of 'Critical' or 'High'. (see above)

      • Resolved: The issue has been resolved/implemented but is either under testing or pending code review/validation from another developer.

      • Closed: The issue has been resolved/implemented and tested - no further work will be done on this issue.

    • Resolution: How the issue was resolved/implemented.

      • By Design: The issue is indeed a design of the system. Only applicable to defects.

      • Cannot Reproduce: The developers cannot reproduce the issue. Only applicable to defects.

      • Code Fix: Changes are made in the source code level. Note that changes to database schema belongs to this category as well.

      • Data Fix: Changes are made to the data in the database. Do not confuse this with a change to the database schema. (see above)

      • Do Not Fix: The developers have agreed the issue is valid but have decided not to make any changes.

      • Duplicate Issue: The issue represents the same problem as a previously reported issue. ;Note that an issue of this type should be resolved and set to 'Duplicate Issue' only after the previously reported issue has been resolved.

      • False Submission: The issue was submitted by mistake/accident.

      • System Fix: Changes are made in the system level.

      • User Error: The issue was a result of the user error or misuse. Only applicable to defects.

    • Priority: How soon and important the issue has to be resolved.

      • For Defects:

        • Critical The defect has no workarounds and the lab cannot operate without the defect being resolved. Usually the defect will have system-wide impact. Must be resolved immediately in the current production release. For example:

          • alDente homepage cannot be loaded.

          • Sample sheets cannot be generated when the lab is about to start a sequence run.

        • HighThe defect has no workarounds but the lab can still operate for a period of time while waiting the defect to be resolved. Usually the defect will only affect a small number of users. Should be resolved ASAP in the current production release. For example:

          • Last 24 Hours page not working; however no one is desperately waiting for the analysis results.

          • Error message showing up during the protocol steps preventing an individual lab user from continuing his/her experiment.

        • MediumThe defect has workarounds and the lab can still operate without the defect being resolved. However, the defect reduces the ease to use the system and thus affects users work productivity in major ways. Should be resolved in the next production release. For example:

          • Lab administrators cannot create new employees - they have to come to the developers in order to fulfill the task.

          • Plates cannot be deleted from the "Delete or Annote Recent Plates" button from the homepage; however, users can still scan in the plates and delete them from the multi-record edit page.

        • LowThe defect has workarounds and the lab can still operate without the defect being resolved. At most, the defect should only affect the users work productivity in minor ways. May be resolved in any future product releases as time permits. For example:

          • Users cannot scan their employee barcode (with password) to quickly relogin. ;Instead, they have to go back to the Re-Login page and login from there.

          • JavaScript text searching for libaries are not working; instead the users need to scroll down the popup menu to find the library they want.

        • For enhancements:

        • CriticalThe enhancement is critical to the operation of the lab and will be implemented in the next production release which needs to be available to the lab ASAP. (i.e. the release date will be earlier than the original scheduled/anticipated release date). Note that this should almost never happen as this introduces sigificant risk into the system. Instead, the developers should always be given advanced notice about any new features that needed to be implemented. For example:

          • Need to be able to generate sample sheets for the new sequencer and the lab needs to be able to use this sequencer in a few days.

        • HighThe enhancement is very important to the operation of the lab and will be implemented in the next production release. For example:

          • A new library submission process for users to submit new libraries.

        • MediumThe enhancement is somewhat important to the operation of the lab and preferrably be implemented in the next production release if possible. For example:

          • Catalog number searching when receiving stock - allowing most information to be pre-filled.

        • LowThe enhancement is the 'nice-to-have' type which can be implemented in any future production releases if time permits. For example:

          • Adding a button to the multi-record edit page allowing batch selecting of records.

        • For maintenance:

        • CriticalDisaster time! The lab is not operating and the maintenance must be performed immediately. For example:

          • MySQL data folder is full and no more queries can be issued against the database.

          HighThe maintenance should be addresses ASAP or else it will soon jeopardize the normal operation of the lab. For example:

          • /home/sequence directory has exceed 90% disk usage.

          MediumThe maintenance is relatively important to the operation of the lab and should be addressed in the near future. For example:

          • Generate a list of missing trace files in /home/sequence and see if they can be recoved from system backups.

          Low

          • Clean up unused tables in the sequence database.

  • Note that although Critical/High issues tend to be Fatal/Major while Medium/Low issues tend to be Minor/Cosmetic, there can be exceptions. For example:

    • A High and Cosmetic defect: A collaborator is looking at the Summary page and realizes the Q20 histogram outrageously big. ;Although this is strictly cosmetic, it can ruin the collaborator's impression on GSC and so the defect can be deemed High or even Critical.

    • A Low and Major defect: Orders cannot be entered through the Orders page. ;Although this affects the functionality of the software in a major way, if the purchasing department is not using this page at all then this defect is just a low priority one.

  • In the Issues Tracker home page, user can search for existing issues and viewing them, submit new issues, or look at various issues statistics:

  • The Issues Tracker interface for submitting new issue and updating new issue allows users to view existing messages and add new messages as well:

  • An email notification will be sent to a developer whenever a new issue is assigned to him/her or an existing issue is re-assigned to him/her.

  • There are now 2 ways to submit a defect:

    • Through the Issues Tracker interface (i.e. the Submit New Issues link).

    • Through the existing Email Notification button.

  • Note that all defects reported through the Issue Tracker interface or the Error Notification button are defaulted to be High and Major. ;Once assigned to the developers, the defect will be triaged and its priority/severity may be changed as needed.
